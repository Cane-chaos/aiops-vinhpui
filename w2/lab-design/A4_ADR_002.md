# ADR-002: Chuẩn hóa Collection bằng OpenTelemetry Collector và Consolidate UI bằng Grafana (Single-Pane-of-Glass)

## Status
**Accepted**

---

## Context

### Vấn đề hiện tại
Kiến trúc hiện tại chạy **2 agent song song** trên mỗi host:
- **Datadog Agent**: Thu thập metrics, traces, logs → gửi tới Datadog SaaS.
- **Splunk Universal Forwarder**: Thu thập logs → gửi tới Splunk Cloud.

On-call engineer phải mở **4 UI khác nhau** khi xử lý incident:
1. Datadog APM (traces + service map)
2. Datadog Logs (hot log search 15 ngày)
3. Splunk Cloud (long-tail log search 30 ngày)
4. PagerDuty (incident routing)

**Hệ quả** (trích từ pain_points.md):
- Pain point #3: Median time "page received → first hypothesis" = **8 phút** do phải switch qua 4 UI.
- Pain point #8: Onboarding 2-3 tuần — mỗi tool có query language, concept model, UI riêng. Engineer mới: "Tôi không biết mở tool nào trước".
- Pain point #9: Không thể query "tất cả incidents service X trong 90 ngày có action=rollback" — vì incident data nằm rải rác.

### Số liệu liên quan
- 29 incidents / 6 tháng. Trung bình 5 incidents/tuần.
- MTTD trung bình: 21.9 phút. MTTR trung bình: 42.3 phút.
- 8/29 incidents (28%) liên quan multi-service cascade → cần correlation cross-signal.
- 3 engineers duy trì alert routing rules thủ công (side responsibility, không dedicated headcount).

---

## Decision

### Phần 1: OpenTelemetry Collector làm lớp Collection chuẩn hóa

Thay thế **cả 2 agents** (Datadog Agent + Splunk Forwarder) bằng **OpenTelemetry Collector** triển khai 2 tầng:

1. **DaemonSet (per-node)**: Thu thập metrics (Prometheus scrape), logs (filelog receiver), traces (OTLP receiver) từ tất cả 10 services trên mỗi node.
2. **Gateway (2 replicas, HA)**: Nhận từ DaemonSet → xử lý (transform, filter, sample, enrich) → export tới backends (VictoriaMetrics, Loki, Tempo).

### Phần 2: Grafana làm Single-Pane-of-Glass

Consolidate **tất cả** query/dashboard/alerting UI vào **1 Grafana instance** (self-hosted, HA):
- **Grafana Explore**: Query metrics (PromQL → VictoriaMetrics), logs (LogQL → Loki), traces (TraceQL → Tempo) trong cùng 1 interface.
- **Exemplars linking**: Click từ metric spike → jump tới related traces → click trace span → jump tới associated logs. Zero context-switch.
- **Grafana Alerting**: Unified alert rule engine query từ cả 3 backends. Thay thế Datadog Monitors + Splunk Alerts.
- **Alertmanager**: Fingerprint grouping + inhibition rules → giải quyết pain point #5 (47 alerts → 4 groups).

---

## Alternatives Considered

### Alt 1: Giữ Datadog Agent + Splunk Forwarder (status quo)
- **Pro**: Không cần migration. Team đã quen.
- **Con**: 2 agents = double resource usage trên mỗi host. 4 UI = cognitive overload. Vendor lock-in sâu (Datadog proprietary agent format).
- **Rejected vì**: Không giải quyết pain point #3 (4 UI), #8 (onboarding), #9 (no queryable history). Không thể giảm cost vì agent tied to vendor billing.

### Alt 2: Consolidate vào Datadog One Platform (bỏ Splunk)
- **Pro**: 1 agent (Datadog), 1 UI. Datadog đã có cross-signal correlation (metrics ↔ logs ↔ traces trong cùng UI).
- **Con**: Datadog pricing per-host → $42K+/tháng (tăng nếu mang Splunk logs vào Datadog). Vendor lock-in sâu nhất. Custom metric cardinality penalty vẫn tồn tại.
- **Rejected vì**: Cost tăng thay vì giảm. Datadog không có tiered retention linh hoạt. Exit cost cực cao (proprietary agent + query language).

### Alt 3: Dùng Datadog Agent với dual-export (Datadog + OSS backends)
- **Pro**: Giữ Datadog Agent (team quen), nhưng export data song song tới OSS backends → build Grafana UI bên cạnh.
- **Con**: Datadog Agent có hạn chế export format (không native OTLP cho tất cả signals). Phải trả Datadog license cho agent. Phức tạp: 3 data paths (Datadog → SaaS, Datadog → OSS, Splunk → SaaS).
- **Rejected vì**: Vẫn phải trả Datadog per-host pricing. Agent vẫn vendor-locked. Nửa nạc nửa mỡ.

### Alt 4: Dùng Grafana Agent (now Grafana Alloy)
- **Pro**: Native Grafana stack. Tích hợp sẵn với Loki, Tempo, Mimir.
- **Con**: Grafana Alloy ít mature hơn OTel Collector. Smaller community. Nếu sau này muốn đổi backend khác Grafana stack → agent cũng phải đổi.
- **Rejected vì**: OTel Collector là CNCF graduated, vendor-neutral, community lớn hơn 10×. Alloy tốt nhưng OTel Collector cho flexibility dài hạn hơn.

### Alt 5 (Chosen): OpenTelemetry Collector + Grafana
- **Pro**: 
  - **Vendor-neutral**: OTel Collector export tới bất kỳ backend nào (Datadog, Splunk, Prometheus, Loki, Tempo, Jaeger, OpenSearch...) chỉ bằng config change.
  - **Processing pipeline**: Filter, transform, sample, enrich tại collection layer → giảm volume trước khi storage.
  - **CNCF graduated**: Industry standard, massive community, mọi vendor đều support.
  - **Grafana single-pane**: Giải quyết pain point #3, #8, #9 cùng lúc.
- **Con**: 
  - OTel Collector config phức tạp hơn Datadog Agent (YAML config vs GUI).
  - Mất Datadog Agent auto-discovery.
  - Cần tự build/maintain Grafana dashboards (Datadog có community dashboards sẵn).

## Consequences

### Positive Consequences

1. **Giải quyết pain point #3**: Single-pane Grafana → on-call không cần switch giữa 4 UI. Ước tính giảm MTTD 5 phút/incident.
2. **Giải quyết pain point #5**: Alertmanager grouping + inhibition → 47 PagerDuty alerts giảm còn ~4 grouped incidents.
3. **Giải quyết pain point #8**: Onboarding giảm từ 2-3 tuần → 3-5 ngày (1 tool thay vì 3, 1 query language family thay vì 3).
4. **Vendor-neutral future**: OTel Collector là abstraction layer → đổi backend (VictoriaMetrics → Mimir, Loki → ClickHouse, Tempo → Jaeger) chỉ cần thay exporter config. Không bao giờ bị lock-in vào bất kỳ backend nào nữa.
5. **Giảm resource trên mỗi host**: 1 OTel Collector DaemonSet thay vì Datadog Agent + Splunk Forwarder → ~30% less CPU/memory per host.
6. **Exemplars linking**: Metric spike → related trace → associated log trong 3 clicks. Capability này **không tồn tại** trong hệ thống cũ (phải copy-paste timestamp giữa 3 tools).
7. **Tail-based sampling**: OTel Collector Gateway thực hiện tail-based sampling: 100% error + slow traces, 5% normal → tăng trace coverage 10× so với 1% head-based hiện tại (pain point #2).

### Negative Consequences

1. **Mất Datadog APM Service Map auto-generated**: Datadog tự build service map từ traces. Grafana có service graph plugin từ Tempo data nhưng ít chi tiết hơn (không có RPS, error rate per edge realtime). **Mitigation**: Tempo metrics generator tạo RED metrics per edge → Grafana service graph. Đủ cho RCA, thiếu cho capacity planning.
2. **Mất Datadog auto-discovery**: Datadog Agent tự detect containers, databases, integrations. OTel Collector cần config receivers thủ công. **Mitigation**: OTel auto-instrumentation cho Java/Python/.NET + Helm chart templates cho standard receivers.
3. **Config complexity tăng**: OTel Collector pipeline config (receivers → processors → exporters) phức tạp hơn GUI-based Datadog setup. **Mitigation**: Gitops approach — config stored in Git, templated bằng Helm, reviewed via PR.
4. **Alert migration effort**: ~100+ Datadog monitors + Splunk alerts cần convert sang Grafana alerting YAML thủ công. Ước tính 2-3 ngày engineer effort. **Mitigation**: Export Datadog monitors via API → write converter script → manual review top 20 critical alerts.
5. **Training cost**: Team cần learn LogQL, PromQL (nếu chưa biết), TraceQL, OTel Collector config, Grafana Alerting. Ước tính 2 ngày training workshop + 2 tuần shadow period. **Mitigation**: Grafana Labs cung cấp free training materials. PromQL ≈ LogQL syntax → học 1 biết 2.

## Cost / Feature / Risk Analysis

| Trục | Đánh giá |
|---|---|
| **Cost** | Loại bỏ Datadog toàn bộ ($22,860/tháng) + Grafana Cloud ($1,050) + giảm Splunk (ADR-001). OTel Collector + Grafana self-hosted = ~$250/tháng compute. Net savings: ~$23,660/tháng. |
| **Feature** | Gain: single-pane correlation, exemplars, alert grouping, tail-based sampling, vendor-neutral. Loss: Datadog APM profiling, auto-discovery, container live view, Splunk SPL. Net: **positive** cho incident response, negative cho deep debugging (profiling). |
| **Risk** | **MEDIUM**. OTel Collector là CNCF graduated (low technical risk). Grafana là de facto standard. Risk chính: alert migration parity — thiếu 1 alert rule = missed incident. Mitigation: 4-tuần dual-run period. |

## Rollback / Exit Plan

### Rollback OTel Collector
- **Fallback**: Re-deploy Datadog Agent via Helm. Agent config stored in Git. Rollback time: <30 phút.
- **Data continuity**: OTel Collector dual-write giai đoạn migration → Datadog vẫn có data đầy đủ trong suốt dual-run period.
- **Trigger**: Nếu OTel Collector drop rate >0.1% sustained hoặc latency overhead >5ms p99 → rollback.

### Rollback Grafana (back to Datadog UI)
- **Fallback**: Datadog UI vẫn accessible trong suốt dual-run period (monthly billing, re-enable bất kỳ lúc nào).
- **Dashboard preservation**: Export tất cả Grafana dashboards → JSON backup. Datadog dashboards vẫn exist (chỉ bị disable, không delete).
- **Alert rollback**: Re-enable Datadog monitors (disabled, not deleted) + Splunk alerts. Rollback time: <1 giờ.

### Exit strategy dài hạn
- **Nếu Grafana stack không phù hợp sau 6 tháng**: OTel Collector cho phép route data tới bất kỳ backend: Datadog, New Relic, Elastic, SigNoz, Coroot. Chỉ cần thay exporter config. **Đây là giá trị cốt lõi của OTel: bạn không bao giờ locked-in nữa.**
