# ADR-001: Use Kafka instead of direct push for observability transport

## Status

Accepted

---

## Context

The AIOps platform for the Payment Service must collect and transport three types of telemetry signals at scale: **metrics** (100K–10M events/sec), **structured logs** (50 GB–5 TB/day), and **distributed traces** (proportional to request volume). The platform needs to route these signals to multiple downstream consumers: metric storage (VictoriaMetrics), log storage (Loki), trace storage (Jaeger), cold archival (S3), and an ML anomaly detection pipeline.

Two architectural options were evaluated for the transport layer:

1. **Direct Push**: The OTel Collector or Fluent Bit writes directly to each storage backend (Prometheus remote write, Loki push API, Elasticsearch bulk API).
2. **Kafka as a Message Bus**: All signals are published to Kafka topics. Downstream consumers (Flink, storage connectors) read from Kafka independently.

The payment domain has strict requirements: signal data must not be lost during downstream outages, the system must support future consumers without modifying the ingestion pipeline, and traffic spikes (flash sales, end-of-day settlements) must be absorbed gracefully.

---

## Decision

We will use **Apache Kafka** (AWS MSK) as the central transport layer between signal collection (OTel Collector / Fluent Bit) and downstream storage/processing systems.

Signals flow as follows:

- OTel Collector publishes to `metrics-topic` and `traces-topic`
- Fluent Bit publishes to `logs-topic`
- Apache Flink consumes all topics, enriches/aggregates, and writes to storage
- The ML pipeline consumes `metrics-topic` independently for real-time anomaly detection

Kafka will be configured with:
- Replication factor: 3 (multi-AZ)
- Retention: 7 days (replay window)
- Consumer groups: separate groups for storage, ML pipeline, and archival

---

## Consequences

### Positive

| Consequence | Detail |
|---|---|
| **Replay on downstream failure** | If Loki or VictoriaMetrics goes down, Kafka retains up to 7 days of messages. After recovery, Flink replays from the last committed offset — **zero data loss** for payment observability data. |
| **Fan-out to multiple consumers** | Storage, ML pipeline, and S3 archiver all read the same topic independently. Adding a new consumer (e.g., a fraud detection pipeline) requires **no changes to the Payment Service or OTel Collector**. |
| **Backpressure absorption** | During traffic spikes (e.g., 10M metric events/sec during flash sales), Kafka acts as a shock absorber. Direct push would overwhelm OpenSearch / Prometheus remote write endpoints and cause retries cascading back to the service. |
| **Exactly-once delivery** | Flink + Kafka support exactly-once semantics via transactional producers and Kafka checkpoints — critical for accurate metric aggregation in financial systems. |
| **Decoupled evolution** | Storage backends can be replaced (e.g., Loki → OpenSearch) without touching the collection layer. Schema evolution is handled at the Flink layer. |

### Negative

| Trade-off | Quantified Impact |
|---|---|
| **Added latency** | Direct push to Prometheus remote write: **<10 ms**. Kafka round-trip (produce + consumer poll interval): **+5–20 ms** added latency. Acceptable for observability signals; not acceptable for synchronous payment processing (which does not use this pipeline). |
| **Operational complexity** | Kafka requires broker management, partition tuning, consumer lag monitoring, mTLS/ACL configuration, and topic lifecycle management. Estimated **0.5–1 FTE** additional operational load at medium scale. |
| **Infrastructure cost** | Medium tier (100 services): Kafka on AWS MSK with 3 brokers (m5.xlarge), multi-AZ, 1 TB storage ≈ **$2,000–$3,000/month** incremental cost versus direct push (which would require larger/more redundant storage clusters instead). |
| **Consumer lag risk** | If Flink falls behind, consumer lag grows. Requires dedicated lag alerting (`kafka_consumer_group_lag > 100K events` triggers PagerDuty). |

---

## Alternatives Considered

### Direct Push (Rejected)

**Pros:**
- Simpler architecture — fewer moving parts
- Lower end-to-end latency: <50 ms total pipeline latency
- No Kafka operational overhead
- Lower cost at very small scale (<10 services)

**Cons:**
- **No replay**: If Loki is down during a payment incident, logs from that window are **permanently lost**
- **No fan-out**: Sending to multiple backends requires parallel HTTP calls from OTel Collector, increasing memory/CPU pressure on the collector
- **Tight coupling**: Changing the storage backend requires reconfiguring every OTel Collector instance
- **Backpressure risk**: OTel Collector's memory queue is finite (~1 GB default). During spikes, oldest telemetry is silently dropped

**Verdict**: Direct push is viable for prototypes or very small deployments (<10 services, <10 GB/day). For a payment system requiring auditability and resilience, the risk of data loss outweighs the simplicity benefit.

### AWS Kinesis (Considered but not selected)

Kinesis is a fully-managed alternative to MSK with lower operational overhead. However:
- Shard-based throughput model is less flexible than Kafka partitions for variable-rate payment traffic
- Consumer library (KCL) is AWS-specific, creating vendor lock-in
- Cost at 10M events/sec exceeds MSK cost by approximately 40%

Kinesis remains a valid choice for AWS-native teams that prefer managed services over operational flexibility.
